# Research: Erro 403 ao deletar clube

**Date**: 2026-05-12
**Purpose**: Investigar a causa raiz do `PATCH /rest/v1/clubs?id=eq.<uuid> 403 (Forbidden)` ao tentar remover um clube na tela de admin.

## Overview

A operação de "deletar clube" no front-end é, na verdade, um **soft delete** — um `UPDATE` em `public.clubs` setando `deleted_at`. O navegador faz `PATCH /rest/v1/clubs?id=eq.<id>`. O retorno 403 do PostgREST indica que a política RLS de UPDATE rejeitou o comando (ou o `GRANT` falhou, mas isso já está concedido).

## Current Implementation

### Key Files

- `js/clubs.js:172-179` — `softDeleteClub(clubId)`: faz `supabase.from('clubs').update({ deleted_at, updated_at }).eq('id', clubId)`. É o ponto que gera o PATCH.
- `js/pages/admin/clubs.js:114-127` — handler do botão `.delete-club-btn` que chama `softDeleteClub` após `confirm()`.
- `migrations/008_clubs_linked_to_students.sql:44-66` — define RLS da tabela `clubs`, inclusive a policy `clubs_update`.
- `js/app.js:199-227` — `loadProfile()` com fallback que usa `user_metadata.role` quando o registro em `public.users` falha/não existe.
- `migrations/002_rls_security.sql:24-46` — RLS de `public.users` (`users_select` requer apenas `auth.uid() IS NOT NULL`).
- `migrations/004_auth_users_profile_trigger.sql` — trigger que insere o registro em `public.users` no signup.

### How It Works

Fluxo do "delete":

1. Admin clica em "remover" → `confirm()` → `softDeleteClub(id)`.
2. Supabase JS emite `PATCH /rest/v1/clubs?id=eq.<uuid>` com body `{ deleted_at, updated_at }`.
3. Postgres avalia a policy `clubs_update`:

```sql
CREATE POLICY clubs_update ON public.clubs
  FOR UPDATE TO authenticated
  USING (
    (SELECT role FROM public.users WHERE id = auth.uid()) = 'admin'
  );
```

4. Se o subselect retorna `'admin'`, o UPDATE passa. Caso contrário, PostgREST devolve **403 Forbidden** (comportamento típico quando nenhuma linha satisfaz USING numa operação modificadora com retorno representational — o PostgREST mapeia para 403 quando há *RLS check failure* explícito; mapeia para 200 com array vazio quando o WHERE não casa).

### Code Patterns

- Soft delete por `deleted_at IS NULL` em todas as queries de leitura (`js/clubs.js:165`, `js/pages/admin/clubs.js:57`).
- RLS em `clubs` é **separada e mais simples** do que o resto do projeto: usa `(SELECT role FROM users WHERE id = auth.uid()) = 'admin'` em vez do padrão usado em `002_rls_security.sql` que faz `EXISTS (... AND role::text = 'admin')`. Funcionalmente equivalente para enum/text — não é o culpado.
- `GRANT SELECT, INSERT, UPDATE ON public.clubs TO authenticated` está concedido (linha 46 da 008). Não há `GRANT DELETE` — mas o front faz UPDATE, então isso é coerente.

## Dependencies

- **Tabela `public.users`**: a policy de update em `clubs` depende de existir uma linha em `users` com `id = auth.uid()` e `role = 'admin'`. Se a linha não existir (ou se `role` estiver com outro valor), o subselect retorna `NULL`, e `NULL = 'admin'` é `NULL` (falsy em USING).
- **Trigger `handle_new_auth_user`** (migration 004): é o que materializa o registro em `users` depois do signup. Contas criadas antes da migration 004, ou via console do Supabase sem passar pelo trigger, podem **não** ter linha em `public.users`.
- **Fallback de perfil em `app.js:200-203`**: quando `select * from users where id = auth.uid()` falha, o app **mente** para a UI dizendo que o role vem de `user_metadata`. Isso permite o usuário enxergar telas de admin (passa no RBAC do front) **mesmo sem ter linha em `users`** — mas o RLS do banco continua negando.

## Important Notes

- **Causa raiz mais provável**: o usuário autenticado **não tem** uma linha em `public.users` com `role = 'admin'`. Cenários possíveis:
  - Conta antiga, criada antes da migration 004 / antes da policy `clubs_update`, sem registro em `public.users`.
  - Conta com `users.role` diferente de `admin` (ex.: `student` ou `responsible`), mas com `auth.users.user_metadata.role = 'admin'` — o front "promove" via fallback (`app.js:201`), o banco não.
  - Migration 008 nunca aplicada nesse projeto Supabase (`clubs_update` ausente → mas nesse caso o erro seria diferente; aqui claramente a policy existe).
- **Observação secundária**: a policy `clubs_update` tem só `USING`, sem `WITH CHECK`. Postgres usa `USING` para ambos. Como o `UPDATE` muda apenas `deleted_at`/`updated_at` (não muda `role`/`created_by`), o `WITH CHECK` implícito continua passando — desde que o usuário seja admin. Não é fonte do 403.
- **Observação secundária 2**: após o soft delete, o row deixa de satisfazer `clubs_select` (`deleted_at IS NULL`). PostgREST por padrão devolve a representação. Quando ela é filtrada por RLS de SELECT pós-UPDATE, o retorno costuma ser 200 com array vazio **ou** 406 — não 403. Isso reforça que o 403 vem da policy de UPDATE em si, não do filtro de SELECT.

## Verificações sugeridas (para a fase de spec/fix)

Rodar no SQL editor do Supabase autenticado como o usuário em questão:

```sql
-- 1. O auth.uid() atual tem linha em public.users?
SELECT id, role FROM public.users WHERE id = auth.uid();

-- 2. user_metadata bate com public.users.role?
SELECT id, role, raw_user_meta_data->>'role' AS meta_role
FROM auth.users WHERE id = '<uid-do-admin>';

-- 3. A policy existe e está como esperado?
SELECT polname, polqual FROM pg_policy WHERE polrelid = 'public.clubs'::regclass;
```

Se (1) retornar 0 linhas ou `role <> 'admin'`, o 403 está explicado.

## Recommendations

Quando partir para o spec/fix, considerar (em ordem):

1. **Corrigir o dado**: garantir que a conta admin tenha `public.users.role = 'admin'`. Um `UPDATE public.users SET role = 'admin' WHERE id = '<uid>';` resolve o caso pontual.
2. **Tornar o RBAC do front menos otimista**: remover o fallback `user_metadata.role` em `app.js:199-227` ou pelo menos não tratá-lo como `'admin'`, para evitar UIs que prometem ações que o banco recusa.
3. **Endurecer a policy** (opcional): trocar para o padrão usado no resto do projeto (`EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND role::text = 'admin')`) — apenas por consistência, não corrige o caso atual.
4. **Mensagem de erro melhor**: no handler de `softDeleteClub` (`js/pages/admin/clubs.js:120-125`), detectar `err.code === '42501'` / status 403 e exibir "Sua conta não tem permissão de admin no banco. Contate o suporte." em vez de só `err.message`.
